# 创建失败项目断点恢复设计

## 目标

项目创建失败后，用户可以进入项目首页查看失败原因并再次生成。恢复不是简单重复调用：系统先审计现有资料，保护人工正文和锁定大纲，从最早失效的依赖开始重建，并在跨模块一致性、派生状态和 RAG 索引全部通过后才把项目改为 `active`。

## 采用方案

采用依赖感知恢复，而不是无条件全量覆盖或只补空表。

依赖顺序为：确认灵感/项目卡 → 世界观与人物 → 大纲与空白正文 → 组织/地图、伏笔、时间线 → RAG 索引 → 一致性门禁。

- 创建失败且没有人工正文、锁定大纲时，原有未通过最终门禁的自动生成资料视为不可信暂存，可清理后依据同一项目卡完整重建。
- 已有人工正文或锁定大纲时，自动恢复必须停止并返回保护原因，不能覆盖作者资料。
- 重建始终读取项目当前保存的目标总字数、题材、视角、读者和确认灵感，不接受调用方重新指定这些配置。

## 后端接口

- 恢复状态接口返回项目状态、各模块数量、失败/缺失项、人工资料保护状态、是否允许恢复和推荐动作。
- 恢复执行接口仅允许 `generation_failed` 或长时间停留在 `creating` 的项目执行，并使用项目数据库中的确认配置。
- 同一项目同时只能存在一个恢复任务；重复请求返回当前任务状态，不启动第二个任务。

## 一致性门禁

恢复完成前必须验证：

1. 世界观、人物、组织、地图、伏笔、时间线均存在且关键字段完整。
2. 大纲章节与正文空壳一一对应，章节顺序和 outlineId 可追踪。
3. 每章目标为 3200–4000，所有章节目标合计严格等于项目目标总字数。
4. 伏笔章节引用处于有效范围，人物引用必须存在。
5. 时间线事件章节引用处于有效范围。
6. 角色、大纲、伏笔 RAG 集合完成当前项目的严格重建。

任一门禁失败，项目保持 `generation_failed`，并保存明确错误和缺失模块；不显示为已完成。

## 前端

项目首页根据真实项目状态和模块统计显示创建恢复面板。失败项目提供“继续生成并修复”按钮、保护原因、进度和结果。流程阶段不再把灵感和大纲硬编码为已完成。

## 测试

- 空壳失败项目可以恢复。
- 部分自动生成资料会先清理再完整重建，不产生重复数据。
- 人工正文或锁定大纲阻止自动覆盖。
- 同项目并发恢复被阻止。
- 一致性或 RAG 失败时不得激活。
- 全部门禁通过后状态变为 `active`，各模块与索引可读取。
